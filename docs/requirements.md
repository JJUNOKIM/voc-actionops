# VOC ActionOps 요구사항 정의

## 1. 사용자 유형

VOC ActionOps는 회사 내부 운영자를 대상으로 한다.

## 2. 사용자 역할

### 2.1 Admin

조직과 사용자를 관리하는 최고 권한 사용자다.

권한:

* 조직 정보 관리
* 사용자 초대
* 사용자 역할 변경
* 모든 데이터셋 조회
* 모든 이슈 조회 및 수정
* 대시보드 조회
* 시스템 설정 관리

### 2.2 PM

고객 피드백을 기반으로 제품 개선 우선순위를 판단하는 사용자다.

권한:

* 피드백 조회
* 이슈 생성 및 수정
* 이슈 우선순위 변경
* 담당자 배정
* 액션 아이템 생성
* 이슈 상태 변경
* 대시보드 조회
* AI 분석 결과 수정

### 2.3 CS

고객 문의와 리뷰를 확인하고 피드백 분류를 보정하는 사용자다.

권한:

* 피드백 조회
* AI 카테고리 수정
* AI 감성 분석 결과 수정
* 피드백 코멘트 작성
* 이슈에 피드백 연결 요청
* 대시보드 일부 조회

### 2.4 Developer

할당된 이슈를 확인하고 처리 상태를 업데이트하는 사용자다.

권한:

* 할당된 이슈 조회
* 이슈 상태 일부 변경
* 기술 코멘트 작성
* 액션 아이템 처리
* 관련 피드백 조회

### 2.5 Viewer

데이터를 조회만 할 수 있는 사용자다.

권한:

* 대시보드 조회
* 피드백 조회
* 이슈 조회

제한:

* 데이터 수정 불가
* 상태 변경 불가
* 액션 생성 불가
* AI 분석 결과 수정 불가

## 3. 핵심 기능 요구사항

## 3.1 인증/인가

사용자는 이메일과 비밀번호로 로그인할 수 있어야 한다.

시스템은 사용자의 역할에 따라 접근 가능한 API와 화면 기능을 제한해야 한다.

필수 요구사항:

* 회원가입 또는 사용자 초대
* 로그인
* JWT 기반 인증
* 역할 기반 권한 제어
* 내 정보 조회
* refresh token rotation 기반 토큰 재발급
* 로그아웃

refresh token 원문은 저장하지 않으며, 재발급에 사용한 토큰의 재사용과 로그아웃 이후 사용을 차단해야 한다.

## 3.2 조직 관리

하나의 조직은 여러 사용자를 가질 수 있다.

필수 요구사항:

* 조직 생성
* 조직 정보 조회
* 조직 사용자 목록 조회
* 사용자 역할 변경
* 조직 단위 데이터 분리

## 3.3 데이터셋 업로드

사용자는 고객 피드백 CSV 파일을 업로드할 수 있어야 한다.

필수 요구사항:

* CSV 파일 업로드
* 컬럼 매핑
* 업로드 전 데이터 미리보기
* 필수 컬럼 검증
* 잘못된 행 검출
* 원본 파일 저장
* 업로드 상태 조회

필수 컬럼 후보:

* content
* rating
* source_type
* created_at

선택 컬럼 후보:

* external_id
* customer_segment
* product_name
* language
* channel
* order_id

## 3.4 피드백 관리

업로드된 피드백을 조회하고 필터링할 수 있어야 한다.

필수 요구사항:

* 피드백 목록 조회
* 피드백 상세 조회
* 키워드 검색
* 기간 필터
* source_type 필터
* rating 필터
* sentiment 필터
* category 필터
* issue 연결 여부 필터

## 3.5 AI 분석

피드백 원문을 기반으로 AI 분석 결과를 생성해야 한다.

분석 항목:

* sentiment
* sentiment_score
* category
* urgency_score
* summary
* confidence_score

필수 요구사항:

* AI 분석 요청
* 분석 상태 조회
* 분석 결과 저장
* 분석 실패 시 재시도
* 분석 실패 사유 저장
* 낮은 confidence 결과 표시

## 3.6 AI 분석 결과 수정

사용자는 AI 분석 결과가 부정확할 경우 직접 수정할 수 있어야 한다.

수정 가능 항목:

* sentiment
* category
* urgency_score
* issue 연결

필수 요구사항:

* 수정 전 AI 값 저장
* 수정 후 사용자 값 저장
* 수정자 저장
* 수정 사유 저장
* 수정 시간 저장

## 3.7 이슈 클러스터링

유사한 피드백은 하나의 이슈로 묶어야 한다.

필수 요구사항:

* 피드백 분석 완료 후 이슈 후보 생성
* 기존 이슈와 유사도 비교
* 유사도가 기준 이상이면 기존 이슈에 연결
* 기준 미만이면 새 이슈 생성
* 이슈와 피드백 연결 관계 저장
* 대표 피드백 표시

## 3.8 이슈 관리

사용자는 생성된 이슈를 확인하고 처리 상태를 관리할 수 있어야 한다.

필수 요구사항:

* 이슈 목록 조회
* 이슈 상세 조회
* 이슈 상태 변경
* 이슈 담당자 배정
* 이슈 우선순위 변경
* 이슈 코멘트 작성
* 관련 피드백 목록 조회
* 대표 원문 피드백 조회

이슈 상태:

* NEW
* TRIAGED
* ASSIGNED
* IN_PROGRESS
* RESOLVED
* MONITORING
* CLOSED

## 3.9 우선순위 계산

시스템은 이슈의 우선순위 점수를 계산해야 한다.

계산 요소:

* 피드백 빈도
* 부정 감성 비율
* 긴급도 평균
* 최근 증가율
* 고객 영향도

우선순위 등급:

* P0
* P1
* P2
* P3

## 3.10 액션 관리

이슈 해결을 위한 액션 아이템을 생성하고 관리할 수 있어야 한다.

필수 요구사항:

* 액션 생성
* 담당자 지정
* 마감일 설정
* 액션 상태 변경
* 액션 완료 처리
* 액션 코멘트 작성

액션 상태:

* TODO
* IN_PROGRESS
* DONE
* CANCELED

## 3.11 대시보드

운영자는 전체 VOC 상태를 대시보드에서 확인할 수 있어야 한다.

필수 지표:

* 전체 피드백 수
* 부정 피드백 비율
* 신규 이슈 수
* P0/P1 이슈 수
* 미해결 이슈 수
* 평균 처리 시간
* 카테고리별 이슈 수
* 최근 증가율 높은 이슈
* TOP 이슈 목록

## 3.12 처리 후 지표 추적

이슈가 해결된 뒤 관련 피드백 변화 추이를 추적해야 한다.

필수 요구사항:

* 해결 전후 관련 피드백 수 비교
* 해결 전후 부정 피드백 비율 비교
* MONITORING 상태에서 재발 여부 확인
* 일별 지표 snapshot 저장

## 4. 비기능 요구사항

## 4.1 성능

* 피드백 목록 조회는 페이지네이션을 적용한다.
* 대시보드 집계는 캐싱 또는 snapshot을 사용한다.
* 대량 CSV 업로드 시 API 요청이 timeout되지 않아야 한다.
* AI 분석 작업은 비동기로 처리한다.

## 4.2 보안

* 인증되지 않은 사용자는 API에 접근할 수 없어야 한다.
* 조직 간 데이터는 분리되어야 한다.
* 역할별 권한 제어가 적용되어야 한다.
* 비밀번호는 해시 처리한다.

## 4.3 안정성

* AI 분석 실패 시 실패 사유를 기록한다.
* 비동기 작업 실패 시 재시도할 수 있어야 한다.
* 실패한 작업은 별도 상태로 관리한다.
* 중복 업로드와 중복 분석을 방지한다.

## 4.4 확장성

* 피드백 출처를 추가할 수 있어야 한다.
* AI 분석 모델을 교체할 수 있어야 한다.
* 검색 엔진을 MySQL 검색에서 OpenSearch로 확장할 수 있어야 한다.
* 메시지 큐를 SQS에서 Kafka로 확장할 여지를 둔다.

## 5. 이번 버전에서 제외할 기능

이번 프로젝트의 첫 버전에서는 다음 기능을 제외한다.

* 실시간 채팅
* 모바일 앱
* 결제 기능
* Kubernetes 배포
* Kafka 기반 스트리밍
* GraphQL
* 완전한 MSA
* 복잡한 관리자 통계 권한
* LangChain 중심 구조
* 실시간 크롤링 자동화

## 6. MVP 완료 기준

MVP는 다음 흐름이 웹에서 실제로 동작하면 완료로 본다.

1. 사용자가 로그인한다.
2. CSV 파일을 업로드한다.
3. 피드백 데이터가 저장된다.
4. AI 분석이 실행된다.
5. 유사 피드백이 이슈로 묶인다.
6. 이슈 우선순위가 계산된다.
7. 사용자가 이슈를 확인한다.
8. 담당자를 배정한다.
9. 액션 아이템을 생성한다.
10. 이슈 상태를 변경한다.
11. 대시보드에서 전체 현황을 확인한다.
